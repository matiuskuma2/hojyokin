# 情報の泉 補助金データ

## 概要
「情報の泉」から取得した補助金・助成金のデータです。
将来的にこれらのデータを`subsidy_cache`に格納し、補助金マッチングに活用します。

```
├── README.md                        # このファイル
```

## 推奨データ抽出アーキテクチャ

### フェーズ1: jGrants API（優先）
現在実装済み。国の補助金データをAPIで取得。

### フェーズ2: 情報の泉（Playwright + Firecrawl）

```
┌─────────────────────────────────────────────────────────────┐
│  別サーバー (AWS Lambda / EC2 / 専用サーバー)                │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. Playwright: 情報の泉にログイン                          │
│     ├─ /search-subsidy?per_page=100 GET                    │
│     ├─ meta[name="csrf-token"] を取得                       │
│     └─ Cookie維持                                           │
│                                                             │
│  2. Playwright: POST /policy/detail ループ                  │
│     ├─ form-urlencoded形式                                  │
│     ├─ JSONの html をパース                                 │
│     └─ 「支援URL」を抽出                                    │
│                                                             │
│  3. Firecrawl: 支援URLの本文取得                            │
│     ├─ 自治体サイトのHTML取得                               │
│     ├─ PDF URLの場合は Vision API へ                       │
│     └─ 補助金詳細情報を抽出                                 │
│                                                             │
└─────────────────────────────────────────────────────────────┘
                          │
                          ▼
┌─────────────────────────────────────────────────────────────┐
│  Cloudflare Workers API                                     │
├─────────────────────────────────────────────────────────────┤
│  POST /api/admin-ops/izumi/ingest                          │
│  └─ subsidy_cache に格納                                    │
└─────────────────────────────────────────────────────────────┘
```

### なぜ Playwright + Firecrawl の組み合わせか

| 処理 | ツール | 理由 |
|------|--------|------|
| ログイン・セッション維持 | **Playwright** | Cookie管理が得意、フォーム操作が安定 |
| CSRF対策ページ取得 | **Playwright** | 動的トークン取得が必要 |
| POST APIループ | **Playwright** | セッション維持が必要 |
| 自治体サイト本文取得 | **Firecrawl** | 静的HTML取得が高速 |
| PDF解析 | **Vision API** | 構造化データ抽出 |

### 注意点

- Firecrawlは「ログインフォーム→セッション維持→POST連打」は苦手
- Playwrightは得意（セッション・Cookie維持が簡単）
- 役割分担が重要

## 活用方法

1. **このCSVの支援URLを直接Firecrawlで取得** → 情報の泉へのログイン不要
2. **source_registry登録**: 新しいsourceとして`izumi`を追加
3. **subsidy_cache格納**: 抽出したデータを正規化して格納

## 6. 補助金データ更新計画

### データソースの分類

| ソース | API有無 | 更新方法 | 優先度 |
|--------|---------|----------|--------|
| **jGrants** | ✅ あり | API定期取得（自動化済み） | 高 |
| **情報の泉** | ❌ なし | Playwright + スクレイピング | 高 |
| **J-Net21** | ✅ RSS | RSS定期取得（実装済み） | 中 |
| **自治体個別** | ❌ なし | 個別対応（手動/半自動） | 低 |

### jGrants との関係

```
┌─────────────────────────────────────────────────────────────┐
│  jGrants API（国の補助金）                                   │
│  ├─ 自動取得: ✅ 実装済み                                   │
│  ├─ 更新頻度: 毎日                                          │
│  └─ subsidy_cache に直接格納                                │
└─────────────────────────────────────────────────────────────┘
              ↓ 一部重複
┌─────────────────────────────────────────────────────────────┐
│  情報の泉（地方自治体の補助金を含む）                        │
│  ├─ 自動取得: ❌ 要Playwright                               │
│  ├─ 価値: jGrantsに無い地方補助金を特定可能                 │
│  └─ 重複: jGrantsと一部重複（国の補助金）                   │
└─────────────────────────────────────────────────────────────┘
```

### 情報の泉データの活用戦略

1. **jGrants未掲載の補助金を特定**
   - 情報の泉の `policy_id` と jGrants の ID をマッピング
   - マッピングできないもの = jGrants未掲載 = 価値が高い

2. **地方自治体独自の補助金をカバー**
   - 情報の泉は市区町村レベルの補助金を多数収録
   - jGrants は国・都道府県レベルが中心

3. **新規補助金の検知**
   - 情報の泉の `policy_id` は新規追加時に増加
   - 定期的に最新 `policy_id` をチェック

### 課題と検討事項（詳細）

#### 1. support_url の種類と対応方法

| URLパターン | 例 | 課題 | 対応策 |
|-------------|-----|------|--------|
| **自治体HTML** | `city.xxx.lg.jp/.../page.html` | ページ構造がバラバラ | サイト別パーサー or LLM抽出 |
| **PDF直リンク** | `city.xxx.lg.jp/.../file.pdf` | 期限切れ後に内容変更 | PDF定期ダウンロード + AI解析 |
| **PDF内の情報** | 要綱、チラシ等 | 年度更新で別ファイルになる | 親ページを監視してPDF URL更新 |

#### 2. PDF問題の詳細

```
┌─────────────────────────────────────────────────────────────┐
│  PDF直リンクの問題                                           │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  例: https://city.xxx.lg.jp/files/r6_hojo_youkou.pdf        │
│                                                             │
│  ❌ 問題点:                                                  │
│  ├─ 年度更新で r7_hojo_youkou.pdf に変わる                  │
│  ├─ PDFの中身だけ見ても「今有効か」不明                      │
│  └─ 古いPDF URLがリンク切れになる                           │
│                                                             │
│  ✅ 対応策:                                                  │
│  ├─ 親HTML（案内ページ）を監視                              │
│  ├─ 親HTMLから最新PDF URLを都度取得                         │
│  └─ PDF内容 + 親HTML内容 の両方をチェック                   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

#### 3. データソース別の取得戦略

| ソース種別 | 取得方法 | 更新頻度 | 精度 | コスト |
|-----------|----------|----------|------|--------|
| **jGrants API** | REST API | 毎日 | ◎ 高 | ○ 低 |
| **RSS/Atom** | フィード取得 | 毎日 | ○ 中 | ○ 低 |
| **自治体HTML** | スクレイピング | 週次 | △ 中 | △ 中 |
| **PDF** | ダウンロード + AI解析 | 月次 | △ 中 | ✕ 高 |

#### 4. Firecrawl の限界と代替案

| ツール | 得意 | 苦手 | 用途 |
|--------|------|------|------|
| **Firecrawl** | 静的HTML取得 | 認証、PDF、構造解析 | 初期データ収集 |
| **Playwright** | 認証、動的JS、フォーム | 大量処理の速度 | ログイン必須サイト |
| **Puppeteer** | 認証、動的JS | 環境構築が複雑 | 高速並列処理 |
| **Jina Reader** | Markdown変換 | 日本語サイト | 本文抽出 |
| **LLM + Vision** | PDF解析、構造理解 | コスト | PDF内容抽出 |

#### 5. 実装優先度の整理

```
優先度: 高 ───────────────────────────────────────────────> 低
 
[jGrants API]  [RSS/J-Net21]  [自治体HTML]  [PDF解析]
```
